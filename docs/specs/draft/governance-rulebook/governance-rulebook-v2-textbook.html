Execution-core · belief engine · 17 rules, six strata

How the daemon thinks, shown thinking.

Every five seconds the daemon discards what it believed and derives a fresh belief base over a partially observable world — seventeen stratified Datalog rules, compiled to SQL, evaluated in order inside one transaction. This page is that program written out as prose it generates itself; the margins show it running. With recording dark the book still reads — the margins simply go quiet.

Preface The problem, and the language

Why rules, not code

This system runs AI coding agents — workers — that carry each ticket through a nine-phase pipeline: triage, research, plan, implement, verify, review, PR, merge, deploy. The daemon keeps dozens in flight at once, and its world is only partially observable: it cannot see inside a worker, only the traces work leaves behind — a status file, a process listing, a fresh commit. Observations lag, go missing, or stop without explanation. Silence is ambiguous: a worker that has written nothing for two minutes may be thinking, or dead.

Judging that world in ordinary procedural code goes badly. The judgments scatter into if-statements, the same fact gets re-derived in five places — and, the deeper flaw, code cannot distinguish absent because I checked from absent because I never looked. So the judgments live here instead: seventeen rules, kept separate from the machinery that acts on them, each one an inspectable sentence whose every conclusion carries its chain of evidence.

Datalog, in one minute. The rules are written in Datalog, a small declarative logic language — Prolog’s tamer cousin (same conclusion :- premises shape, but restricted so evaluation always terminates). Capital letters are variables: read T as a ticket, P as the pipeline phase it is in. Predicates named obs_* are raw observations (obs_signal is the status file a worker last wrote, obs_agent the live process listing); everything else is a conclusion some rule derived. Where TypeScript says how — fetch this, loop over that, return early here — a rule says only what must be true:

wedged_never_started(T,P) :- session_registered(T,P), ¬turn_started(T,P), older_than(2m).

“A worker on ticket T in phase P is wedged if it registered, never started its first turn of work, and two minutes have passed.” The engine finds every (T,P) satisfying those conditions at once: no ordering, no early returns, no hidden state — nowhere for a judgment to hide.

Conclusions then build on conclusions. Ground rules read only observations; each later layer may build on — and, crucially, negate — the layers beneath it. (These six reasoning strata are not the nine pipeline phases: the pipeline is the world being watched; the strata are the layers of thought about it.) ¬turn_started is an honest claim only because the layer that derives turn_started is already complete when it is checked — absence as evidence, never oversight.

Is a logic engine extra machinery? The opposite: the complexity lives in the problem — distributed, asynchronous, partially observable — and the rules contain it. Seventeen sentences replace a thicket of conditionals; the same observations always derive the same beliefs; evaluation cannot loop forever; and because a rule states conditions rather than steps, the premises that held are the why — the explanation is the computation, not logging bolted on afterward. Each rule compiles to a single SQL statement the engine executes verbatim: flip any card below to confirm that the sentence you just read is the program that ran.

Stratum 1 Ground correlations — reads observations only

Ground correlations

Nothing in this stratum judges. Four rules correlate raw observations into facts about workers; because no rule here negates anything, they may run in any order — and because they run first, every later stratum may safely negate them.

Stratum 4 Escalation ladder — negation over intent

Escalation ladder

The daemon detects; the agent interprets; the human is a conclusion of last resort. The negation here runs over the intent table rather than over beliefs — cooldowns and attempt budgets are facts, so storms and retry loops are unrepresentable rather than discouraged.

The phase the procedural scheduler would dispatch next — a prediction, never a dispatch. Arm a takes the verify → remediate detour while budget remains; arm b walks the normal FSM edge. The shadow comparator logs any disagreement with the live oracle; it acts on nothing.

    next_phase(P, Next), Next != done, no signal for (A, Next) yet.

Appendix B How to read this page

The vocabulary, in six words

Belief. A conclusion the engine derived this tick, stored as a row with full provenance — the rule that fired and every fact and belief it consumed. Beliefs are re-derived fresh every five seconds; nothing is remembered, everything is re-earned.

Entity. What a belief is about — a ticket-phase (CTL-722/plan), a ticket (CTL-722), or a host (host:mini). The storage column and the API keep their old name, subject (?subject= still works); this page retires the word because in logic it has no stable meaning.

Because (premises). The rule’s body — the conditions that all held. Every derivation on this page reads as a sentence: concluded X about E because…, followed by the premises, one per line.

¬ Checked, absent. Stratified negation: an absence the engine explicitly went looking for and documented — never an absence it failed to notice. Negative premises carry the same weight as positive ones; strata exist precisely so that a rule only negates a set that is already complete.

Threshold. A cfg fact — a live number with a humane reading, listed in Appendix A. Rules compare against thresholds by joining the cfg table, so retuning the daemon is a config edit with provenance, not a deploy.

Concludes / derives. The rule’s head — what it asserts when the premises hold. Some rules derive a value ({free_slots: 4}); for others the existence of the belief is itself the verdict, written here as derives no value — its existence is the verdict.

Stratum. The evaluation order. S1 inserts before S2 reads it, and so on through S6 — which is the entire reason the ¬ above is sound. Hairlines in this book appear only at stratum boundaries, because they are the only boundaries that matter.
